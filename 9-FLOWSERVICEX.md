9-FLOWSERVICEX — Машина состояний и единая шина уведомлений измерительного стола

## 1. Назначение документа
Настоящий документ определяет:
- требования к единой шине уведомлений `TWorkTable.Notify`;
- описание архитектуры доставки событий;
- описание машины состояний измерительного стола `TWorkTable`;
- правила взаимодействия UI и доменной модели в системе FlowServiceX.

Документ объединяет требования (ТЗ) и текущее состояние реализации.

---

## 2. Контекст системы
`TWorkTable` является центральным доменным объектом измерительного стола и отвечает за:
- текущее состояние стенда (`State`);
- управление процессом измерения;
- агрегирование параметров:
  - расход (`FlowRate`);
  - температура (`FluidTemp`);
  - давление (`FluidPress`);
  - насос (`ActivePump`);
- работу с каналами:
  - эталонные (`EtalonChannels`);
  - поверяемые (`DeviceChannels`);
- управление текущей точкой (`CurrentPoint`);
- контроль критериев остановки.

Для исключения прямой связности UI и внутренних компонентов используется единая шина событий:
`TWorkTable.Notify(Event: Integer; Data: TObject = nil)`

---

## 3. Единая шина уведомлений (Notify)
### 3.1. Назначение
Обеспечить централизованную доставку всех событий стола в UI:
- изменения состояния;
- изменения параметров процесса;
- события измерения;
- изменения вложенных объектов;
- управляющие действия.

---

### 3.2. Контракт
Подписка реализуется через интерфейс наблюдателя:
- `Subscribe(const AObserver: IEventObserver);`
- `Unsubscribe(const AObserver: IEventObserver);`

Единая точка обработки:
- `OnNotify(Sender: TObject; Event: Integer; Data: TObject);`

---

### 3.3. Правила использования
1. UI подписывается только на `TWorkTable`.
2. Внутренние объекты не уведомляют UI напрямую.
3. Все события агрегируются через `Notify`.
4. `Event` соответствует значениям `ENotifyEvent`.
5. `Data` содержит ссылку на источник или контекст.
6. Обработка выполняется типобезопасно.
7. Старые обработчики допускаются только временно.
8. Управляющие команды передаются через механизм действий (`EActionWorkTable`).

---

## 4. Архитектура доставки событий
### 4.1. Поток события
1. Изменяется внутренний объект (например, `TFlowRate`).
2. `TWorkTable` формирует событие.
3. Вызывает `Notify`.
4. Подписчики получают событие через `OnNotify`.
5. UI выполняет обновление или действие.

---

### 4.2. Формат событий
Используется единый enum:

```pascal
type
  ENotifyEvent = (
    neUnknown = 0,
    neStateChanged,
    neDataChanged,
    neActionRequested,
    neError
  );
```

Формат передачи:

| Event | Data |
|---|---|
| neStateChanged | TWorkTable |
| neDataChanged | TFlowRate / TPump / др. |
| neActionRequested | Объект действия |
| neError | Exception / TObject |

---

## 5. Реализация в текущей версии
### 5.1. Подписка на столы
В `TMainForm` реализовано:
- автоматическая подписка на все активные столы;
- автоматическая отписка от удаленных;
- очистка подписок при уничтожении формы.

---

### 5.2. Единая точка обработки
Все события обрабатываются через:
`TMainForm.OnNotify(Sender, Event, Data)`

Формат логирования:
- `Event=<код>`
- `Source=<класс источника>`

---

### 5.3. Управление жизненным циклом
При уничтожении формы:
- выполняется отписка от всех столов;
- очищается список наблюдателей;
- предотвращаются утечки.

---

## 6. Машина состояний измерительного стола
### 6.1. Состояния стола
```pascal
type
  EStateWorkTable = (
    swtNONE = 0,
    swtSTANDBY,
    swtCONNECTED,
    swtSTARTMONITOR,
    swtSTARTMONITORWAIT,
    swtMONITOR,
    swtSTOPMONITOR,
    swtCONFIGED,
    swtSTARTTEST,
    swtSTARTWAIT,
    swtEXECUTE,
    swtSTOPTEST,
    swtSTOPWAIT,
    swtCOMPLETE,
    swtFINALREAD,
    swtFAILURE
  );
```

---

### 6.2. Переходы состояний
`swtNONE`
→ `swtSTANDBY`
→ `swtCONNECTED`
→ `swtSTARTMONITOR`
→ `swtSTARTMONITORWAIT`
→ `swtMONITOR`
→ `swtSTARTTEST`
→ `swtSTARTWAIT`
→ `swtEXECUTE`
→ `swtSTOPTEST`
→ `swtSTOPWAIT`
→ `swtCOMPLETE`
→ `swtFINALREAD`

---

### 6.3. Публикация состояния
При каждом изменении состояния:
`Notify(neStateChanged, Self);`

---

### 6.4. Основная логика `swtEXECUTE`
На каждом цикле:
1. Обновляются сигналы каналов.
2. Вычисляется `CurrentImp`.
3. Вычисляется `CurrentVolume`.
4. Проверяются критерии остановки:
   - время;
   - импульсы;
   - объем.
5. При выполнении условий:
   - переход в `swtSTOPTEST`.

---

### 6.5. Действия (`EActionWorkTable`)
#### Назначение
Действия используются для передачи управляющих команд через `Notify`.

---

#### Перечисление действий
```pascal
type
  EActionWorkTable = (
    awtNone = 0,
    awtStartTest,
    awtStopTest,
    awtStartMonitor,
    awtStopMonitor,
    awtClampTable,
    awtUnClampTable
  );
```

---

#### Передача действий
`Notify(neActionRequested, ActionObject);`

---

#### Обработка действий в UI
```pascal
case ENotifyEvent(Event) of

  neStateChanged:
    HandleState(Sender);

  neActionRequested:
    HandleAction(Data);

end;
```

---

## 7. Интеграция Notify и машины состояний
Система разделяет три уровня:
- Состояние — `EStateWorkTable`
- Событие — `ENotifyEvent`
- Действие — `EActionWorkTable`

Модель работы:
- state machine изменяет состояние;
- изменения публикуются через `Notify`;
- UI реагирует на события;
- управляющие команды передаются через действия.

## 10. Рекомендации по развитию
- использовать единый `ENotifyEvent`;
- формализовать объект передачи действий;
- публиковать `Notify` при каждом изменении состояния;
- перевести весь UI на `Subscribe/OnNotify`;
- удалить legacy-обработчики после миграции.

---

## 11. Краткое резюме
В системе FlowServiceX реализована единая событийная модель на базе `TWorkTable.Notify`, объединяющая:
- состояния (`EStateWorkTable`);
- события (`ENotifyEvent`);
- действия (`EActionWorkTable`).

Это обеспечивает:
- предсказуемое поведение системы;
- масштабируемость;
- снижение связности;
- упрощение разработки и сопровождения UI.
